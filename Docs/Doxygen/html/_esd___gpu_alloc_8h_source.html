1/*
2
3Allocation mechanism for RAM_G.
4
5Allocation returns a handle, consisting part of the allocation index and a sequence code to invalidate the allocation.
6Handle can be turned into an address using Esd_GpuAlloc_Get. This call should be repeated each frame,
7since the allocation may be invalidated by a graphics device reset, or the allocation may be moved by a memory
8defragmentation process.
9Supports up to MAX_NUM_ALLOCATIONS simultaneous allocations.
10
11The regular use pattern is to call Esd_GpuAlloc_Get using a stored handle during the render stage whenever an image is required.
12This function may be called with an invalid handle. Valid handles may become invalid between render stages when not in use.
13Addresses may thus become invalid between frames, and must not be stored. When the handle is invalid the returned address will be GA_INVALID.
14Esd_GpuAlloc_Alloc is then to be used with GA_GC_FLAG to allocate or re-allocate memory space, the new handle must be stored for later use,
15and Esd_GpuAlloc_Get must be called again with the new handle to get the new address in graphics memory.
16
17When not using the GA_GC_FLAG, you must call Esd_GpuAlloc_Free manually when the memory is no longer in use.
18Either allocation option does not guarantee that memory will remain persistently allocated. Graphics memory can be reset whenever necessary.
19
20*/
